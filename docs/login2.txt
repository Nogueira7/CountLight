# 📘 Relatório Técnico – Backend Ecovolt (Versão Atualizada)

## 1. Introdução

Este relatório descreve de forma detalhada todo o trabalho realizado no desenvolvimento e **aperfeiçoamento do backend do projeto Ecovolt**, no âmbito do projeto final de licenciatura.

O foco principal foi a construção de um backend **robusto, seguro, modular e escalável**, seguindo **boas práticas profissionais** de desenvolvimento web moderno.
Ao longo do desenvolvimento, o sistema evoluiu de uma implementação inicial simples para uma arquitetura mais madura, com separação clara de responsabilidades, autenticação profissional e mecanismos de segurança adicionais.

O backend foi desenvolvido em **Python**, utilizando o framework **FastAPI**, com autenticação baseada em **JWT (JSON Web Tokens)**, validação de dados com **Pydantic**, acesso isolado à base de dados **MySQL**, logging estruturado e controlo do estado dos utilizadores.

A estrutura atual e final do backend é a seguinte:


### Objetivo da Estrutura

Esta organização segue princípios de arquitetura limpa:

* **Routes** → apenas recebem pedidos HTTP
* **Services** → lógica de negócio
* **Repositories** → acesso à base de dados
* **Core** → segurança, configuração e logging
* **Models** → validação e tipagem de dados
* **DB** → ligação à base de dados

Esta separação melhora:

* legibilidade
* manutenção
* testabilidade
* escalabilidade futura

## 5. Autenticação e Segurança

### 5.1 Hashing de Passwords

As passwords **nunca** são armazenadas em texto simples.

* Algoritmo: **bcrypt**
* Biblioteca: `passlib`

Funções:

* `hash_password(password)`
* `verify_password(password, password_hash)`

Isto protege contra:

* ataques de força bruta
* fuga de dados da base de dados

---

### 5.2 JSON Web Tokens (JWT)

O sistema utiliza **dois tipos de tokens**:

#### 🔑 Access Token

* Curta duração
* Usado para autenticar pedidos protegidos
* Contém:

  * `sub` (ID do utilizador)
  * `exp`
  * `type = "access"`

#### 🔄 Refresh Token

* Longa duração
* Usado para renovar access tokens
* Contém:

  * `sub`
  * `exp`
  * `type = "refresh"`

---

### 5.3 Middleware de Autenticação

A função `get_current_user`:

* valida o token JWT
* bloqueia refresh tokens
* extrai o ID do utilizador
* protege rotas privadas

Usada com:

```python
Depends(get_current_user)
```

## 7. Logs Estruturados

Foi introduzido **logging estruturado**, em vez de `print`.

### Exemplos de logs:

* Tentativas de login
* Logins falhados
* Registos de utilizadores
* Acessos bloqueados

Os logs aparecem:

* no terminal durante desenvolvimento
* prontos para ficheiro ou sistemas externos no futuro

Isto permite:

* auditoria
* deteção de abuso
* diagnóstico de erros

---

## 8. Rotas de Autenticação

Localizadas em `routes/auth.py`:

### `/login`

* valida credenciais
* verifica estado do utilizador
* gera access + refresh token

### `/register`

* valida dados
* normaliza username
* cria utilizador

### `/me`

* rota protegida
* retorna ID do utilizador autenticado

### `/refresh`

* valida refresh token
* gera novo access token

### `/logout`

* invalida refresh token na base de dados

---

## 9. Validação de Dados (Pydantic)

Implementada em `models/user.py`:

* Username:

  * normalizado (lowercase, trim)
  * regex restrita
* Email:

  * validado automaticamente
* Password:

  * mínimo de caracteres
  * maiúsculas, minúsculas e números

Benefícios:

* segurança
* dados consistentes
* erros claros
* documentação automática via Swagger

---

## 10. Acesso à Base de Dados

O acesso à base de dados foi isolado em **repositories**:

### `user_repository.py`

* `get_user_by_username`
* `create_user`
* `save_refresh_token`

Benefícios:

* SQL fora das rotas
* fácil migração futura
* código mais limpo

---

## 11. main.py e Orquestração

O ficheiro `main.py` tem apenas:

* criação da app FastAPI
* configuração CORS
* inclusão de routers

Não contém:

* lógica de negócio
* SQL
* autenticação direta

Isto representa uma arquitetura correta e profissional.
