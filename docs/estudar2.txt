# 📘 RESUMO GERAL – PROJETO ECOvolt (Estado Atual)

## 🎯 Objetivo do Projeto

O projeto **CountLight** tem como objetivo o desenvolvimento de uma **plataforma completa de monitorização energética**, capaz de:

* recolher dados reais de consumo elétrico (IoT – Shelly Plug S)
* armazenar leituras energéticas de forma persistente
* gerir utilizadores e autenticação segura
* fornecer dados ao frontend para dashboards energéticos
* manter uma arquitetura modular, escalável e profissional

O sistema foi desenhado com **separação clara de responsabilidades**, permitindo evolução futura sem refatorações profundas.

---

## 🧩 VISÃO GERAL DA ARQUITETURA

O projeto está dividido em **três grandes blocos independentes**:

```
[ Shelly Plug S ]
        ↓ MQTT
[ Mosquitto (Docker) ]
        ↓
[ Data Ingestion (Python) ]
        ↓
[ MySQL (Docker) ]
        ↑
[ Backend FastAPI ]
        ↑
[ Frontend ]
```

📌 Cada bloco é independente, comunicando apenas através de protocolos e dados bem definidos.

# 📁 DATA INGESTION – INJEÇÃO DE DADOS ENERGÉTICOS

## 🎯 Objetivo

O módulo **data_ingestion** é responsável por:

* recolher dados enviados pelos dispositivos Shelly
* processar mensagens MQTT
* validar dados recebidos
* gravar leituras energéticas na base de dados

📌 Este módulo **não é uma API** — é um **serviço de backend em background**.

---

## 🔧 Tecnologias Usadas

* Python 3.10
* paho-mqtt
* mysql-connector-python
* python-dotenv
* logging

---

## 📁 Estrutura Base

```
data_ingestion/
 ├── main.py
 ├── mqtt_client.py
 ├── db.py
 ├── config.py
 ├── utils.py
 ├── .env
```

---

## 🔁 Fluxo de Ingestão de Dados

```
Shelly Plug S
 → envia dados MQTT
 → Mosquitto recebe
 → mqtt_client subscreve tópicos
 → parse da mensagem
 → validação de dados
 → inserção no MySQL
```

---

## ⚙️ Configuração via `.env`

O módulo **data_ingestion** lê diretamente do seu `.env`:

```env
# MQTT
MQTT_BROKER=127.0.0.1
MQTT_PORT=1883

# MYSQL
DB_HOST=localhost
DB_USER=energia_user
DB_PASSWORD=energia_pass
DB_NAME=countlight_db
DB_PORT=3306
```

📌 Isto permite:

* trocar de base de dados facilmente
* mover o projeto para outro servidor
* manter segredos fora do código

---

## ⏱️ Controlo de Inserções

* existe um intervalo mínimo entre gravações
* evita spam de dados
* protege a base de dados
* simula comportamento real de sistemas industriais

---

# 🔁 BACKEND FASTAPI – FUNÇÃO NO SISTEMA

O backend **não recolhe dados diretamente do Shelly**.

Ele:

* lê dados já persistidos no MySQL
* aplica regras de negócio
* expõe endpoints REST
* fornece dados ao frontend

📌 Isto garante baixo acoplamento e alta escalabilidade.
