# 📘 RESUMO GERAL – BACKEND ECOvolt

## 🎯 Objetivo do Backend

O backend do projeto **CountLight** tem como objetivo:

* gerir autenticação e utilizadores
* garantir segurança (passwords, tokens, permissões)
* fornecer dados ao frontend (dashboards energéticos)
* servir como base para integrar dados do **Shelly Plug S**
* manter uma arquitetura limpa, escalável e segura

---

# 🧰 FERRAMENTAS & TECNOLOGIAS UTILIZADAS

## 🐍 Linguagem

### **Python 3.10**

* Linguagem principal do backend
* Grande ecossistema
* Excelente para APIs, segurança e integração IoT

---

## ⚡ Framework Backend

### **FastAPI**

Usado para:

* criar a API REST
* definir endpoints (`/login`, `/register`, etc.)
* validar dados automaticamente
* gerir dependências (`Depends`)
* lidar com HTTP status codes

📌 Vantagens:

* rápido
* moderno
* tipagem forte
* documentação automática (Swagger)

---

## 🔐 Segurança & Autenticação

### **JWT (JSON Web Tokens)** – via `python-jose`

Usado para:

* autenticação stateless
* proteger rotas
* identificar utilizadores

Tipos:

* **Access Token** → curto prazo, acesso às rotas
* **Refresh Token** → longo prazo, renovar sessão

---

### **OAuth2PasswordBearer**

Usado para:

* extrair tokens automaticamente do header
* padronizar autenticação Bearer
* integrar bem com FastAPI

---

### **Passlib + Bcrypt**

Usado para:

* fazer hash seguro de passwords
* verificar passwords sem nunca guardar texto plano

📌 Bcrypt é lento por design → proteção contra brute-force.

---

## 🗄️ Base de Dados

### **MySQL**

Usado para:

* armazenar utilizadores
* guardar refresh tokens
* (futuro) guardar dados energéticos

📌 Comunicação feita manualmente com `cursor` (SQL explícito).

---

## 📦 Outras Bibliotecas Importantes

### **python-dotenv**

* Carrega variáveis de ambiente do `.env`
* Evita segredos hardcoded

---

### **logging**

* Registo de eventos importantes
* Auditoria de login/registo
* Debug e análise em produção

# 🏗️ ARQUITETURA DO BACKEND (CONCEITO CHAVE)

O backend segue uma **arquitetura em camadas**:

```
Routes → Services → Repositories → Database
              ↑
            Core
```

### Porquê isto é importante?

* código organizado
* fácil de manter
* fácil de testar
* escalável
* profissional

---

# 📁 ESTRUTURA DE PASTAS – EXPLICAÇÃO COMPLETA

## `backend/app/main.py`

### 🎯 Objetivo

* ponto de entrada da aplicação
* cria a app FastAPI
* configura CORS
* liga os routers

### Responsabilidades

* ❌ não tem lógica de negócio
* ❌ não fala com a BD
* ✅ apenas configuração global

---

## `backend/app/routes/`

### 🎯 Objetivo

* definir **endpoints HTTP**
* receber pedidos do frontend
* devolver respostas HTTP

### Características

* valida dados de entrada (Pydantic)
* injeta dependências (`Depends`)
* chama services
* não contém lógica pesada

### Ficheiros

#### `auth.py`

Endpoints:

* `POST /login`
* `POST /register`
* `GET /me`
* `POST /refresh`
* `POST /logout`

---

## `backend/app/services/`

### 🎯 Objetivo

* conter a **lógica de negócio**
* decidir regras do sistema

### Características

* valida credenciais
* decide quando falhar
* cria tokens
* coordena repositories
* lança erros HTTP

### Ficheiros

#### `auth_service.py`

Funções principais:

* `login_user`
* `register_user`

---

## `backend/app/repositories/`

### 🎯 Objetivo

* acesso direto à base de dados
* execução de SQL

### Características

* não valida regras
* não conhece FastAPI
* não gera tokens
* só executa queries

### Ficheiros

#### `user_repository.py`

Funções:

* `get_user_by_username`
* `get_user_by_email`
* `create_user`
* `save_refresh_token`
* `get_refresh_token`
* `clear_refresh_token`

---

## `backend/app/core/`

### 🎯 Objetivo

* funcionalidades transversais
* usadas por todo o backend

---

### `security.py`

Responsável por:

* hash e verificação de passwords
* criação de JWT
* validação de tokens
* proteção de rotas

Funções-chave:

* `hash_password`
* `verify_password`
* `create_access_token`
* `create_refresh_token`
* `get_current_user`

---

### `config.py`

Responsável por:

* carregar variáveis de ambiente
* garantir que segredos existem
* centralizar configurações

Variáveis:

* `SECRET_KEY`
* `ALGORITHM`
* `ACCESS_TOKEN_EXPIRE_MINUTES`
* `REFRESH_TOKEN_EXPIRE_DAYS`

---

### `logging.py`

Responsável por:

* configurar logs globais
* criar logger do projeto

---

## `backend/app/db/`

### 🎯 Objetivo

* criar ligação à base de dados
* gerir sessões/conexões
* injetar BD nos endpoints

---

## `__init__.py`

### 🎯 Objetivo

* indicar ao Python que a pasta é um package
* permitir imports organizados

📌 Pode estar vazio — é normal.

---

## `__pycache__/`

### 🎯 Objetivo

* cache automática do Python
* melhora performance

📌 Nunca se edita nem versiona.

# 🔁 FLUXOS IMPORTANTES (PARA MEMORIZAR)

## 🔐 Login

```
Frontend
 → POST /login
 → Route (auth.py)
 → Service (auth_service.py)
 → Repository (user_repository.py)
 → MySQL
 → JWT access token
```

---

## 📝 Registo

```
Frontend
 → POST /register
 → valida username/email
 → hash da password
 → INSERT user
```

---

## 🔒 Rota protegida

```
Request
 → Authorization: Bearer <access_token>
 → get_current_user
 → valida JWT
 → devolve user_id
```
